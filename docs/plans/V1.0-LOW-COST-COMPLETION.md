# Low-cost plan from RC to stable v1.0

Status: ready to execute after owner approval

This is the operational addendum to the approved
[`V1.0-MASTER-PLAN.md`](V1.0-MASTER-PLAN.md). It does not change the frozen measurement rules or
release gates. It changes who does the remaining work so routine processing uses the cheapest
capable model and expensive review happens once.

## What is actually left

The P02–P32 implementation is already assembled in the release candidate. Stable v1.0 needs one
consolidated evidence campaign:

1. run the development videos and compare the original with the candidate;
2. compute the speed, reliability, duplicate-removal, token, event-recall, and answer-quality
   results;
3. validate the answer grader against blinded human judgments;
4. freeze one candidate and verify the confirmation set is still sealed;
5. after explicit human authorization, open that set once and run the confirmation evaluation;
6. audit the claims, docs, install surfaces, artifact, and stable release.

No new feature is added during this campaign unless a measured release blocker requires it.

## Cheapest-capable routing

| Role | Model | Use it for | Do not use it for |
| --- | --- | --- | --- |
| Mechanical worker | Luna (`gpt-5.6-luna`) | inventories, hashes, manifests, running fixed commands, collecting JSON/JSONL, computing deterministic summaries, updating status tables | changing algorithms, interpreting ambiguous failures, approving claims |
| Engineering worker | Terra (`gpt-5.6-terra`) | bounded fixes, benchmark orchestration, test failures, result interpretation when deterministic rules are insufficient | routine successful runs, final approval of its own work |
| Final reviewer | Sol (`gpt-5.6-sol`) | one adversarial review after the candidate and all evidence are frozen | packet-by-packet reviews, mechanical edits, repeated test watching |
| Decision owner | Human | blinded answer judgments, permission to open the sealed set, public-visibility and stable-release decisions | routine command execution |

Target routing is roughly 70–80% Luna, 15–25% Terra, and one short Sol review. These are workload
targets, not measured price claims. Luna is already available locally as `gpt-5.6-luna`; Terra and
Sol remain available as `gpt-5.6-terra` and `gpt-5.6-sol`.

## Execution sequence

### Pass 1 — development benchmark

- Luna validates the corpus manifest, source hashes, tool versions, and empty output directories.
- Luna runs the fixed original/candidate commands and writes raw receipts without reading full
  transcripts or frames into model context.
- Deterministic programs compute metrics and a compact failure summary.
- Terra sees only failures and the summary. Terra makes one bounded repair pass if a release gate
  fails, then Luna reruns only the affected slice plus the final aggregate check.

Output: multi-video raw results, Pareto table, failures, and reproducible commands.

### Pass 2 — grader validation

- Luna produces position-swapped, identity-hidden answer packets.
- Humans score the packets using the frozen rubric.
- Luna computes agreement and calibration with the repository evaluator.
- Terra investigates only rubric/schema failures or insufficient agreement.

Output: human judgments, agreement report, and a go/no-go result for automated answer scoring.

### Pass 3 — freeze and confirmation

- Luna inventories and hashes the exact candidate, configs, tools, corpus registry, and grader.
- Terra verifies that development work did not expose or alter the sealed confirmation cohort.
- Work pauses only for the owner's explicit authorization to open the sealed set.
- Luna runs the single fixed confirmation command; Terra intervenes only for an infrastructure
  failure covered by the preregistered retry policy. No tuning follows confirmation results.

Output: immutable candidate receipt and the one confirmatory result set.

### Pass 4 — stable release

- Luna generates the claim table directly from measured data and updates README, site, changelog,
  limitations, status, issues, and release notes.
- Terra runs the full deterministic suite, install lifecycle, bundle verification, and stable-tag
  rehearsal; it repairs only concrete blockers.
- Sol performs one read-only adversarial audit of the exact staged tree and evidence. Any P0/P1
  finding gets one Terra repair pass and a focused Sol recheck of that finding only.
- The owner decides whether to create the stable tag and whether to make the repository public.

Output: stable-ready tree, final audit, and release artifact.

## Token and time controls

- Agents receive the relevant packet and machine summary, not the whole master plan or conversation.
- Raw videos, frames, transcripts, and passing logs stay on disk; models see paths, hashes, metrics,
  and failing excerpts only.
- Deterministic validators run before any model escalation.
- Luna handles every successful mechanical path. Terra is invoked only by a named failure class.
- Sol reviews once at the claim/release boundary, not after P02–P32 individually.
- One implementation pass and one consolidated verification campaign replace repeated packet loops.
- Independent work may run in parallel only when output directories and owned files are disjoint.

## Stop and escalation rules

- Infrastructure failure: Luna applies the frozen retry policy once; then Terra diagnoses.
- Test or schema failure: Terra repairs the smallest owned slice; Luna reruns focused checks.
- Measurement misses a promotion threshold: record the result and keep the feature optional or
  rejected. Do not tune against confirmation data.
- Grader disagreement: require human scoring; do not ask Sol to manufacture a consensus.
- Claim ambiguity, privacy risk, provenance concern, or confirmation-protocol drift: stop for Sol
  and, where required, the owner.
- Repository visibility never changes without a separate explicit owner instruction.

## Definition of done

Stable v1.0 is ready only when the multi-video development results, validated human/grader result,
single sealed confirmation run, exact artifact, install checks, limitations, attribution, and one
final Sol audit all agree. Passing tests alone do not authorize a performance claim.
