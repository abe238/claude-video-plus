# claude-video-plus v1.0 master plan

Status: **PROPOSED — implementation requires owner approval**

Date: 2026-07-11

Control: unmodified upstream `bradautomates/claude-video` commit `83da59f`

Candidate: `abe238/claude-video-plus`, beginning from `b90ae21`

This is the proposed canonical execution plan for v1.0. After approval, update
`AGENTS.md`, `docs/ARCHITECTURE.md`, `CONTEXT.md`, and ADR references to point here;
retain the two older evidence-backed plans as historical research and review records,
not instructions that must be mentally merged with this plan.

No feature work is authorized by this document until the owner approves it.

## 1. Outcome

Ship the easiest-to-install, most reliable version of `/watch` while minimizing total
reader context and preserving or improving answer quality. The default installation
remains self-contained standard-library Python plus FFmpeg and yt-dlp. Every additional
runtime, model, account, cookie source, or local server is optional, explicitly selected,
measured, and fail-open.

v1.0 should provide:

- resilient public and authenticated video acquisition;
- native, sidecar, local, and cloud transcription through one deep Interface;
- range-aware transcription that does not process irrelevant audio;
- safe resume and reuse without silently retaining private media;
- portable, verifiable Evidence manifests and bundles;
- question-aware evidence compilation with an executable Control fallback;
- hybrid lexical-semantic retrieval that understands paraphrases and concepts without
  sacrificing exact names, numbers, timestamps, or zero-dependency operation;
- dependency-free FFmpeg and standard-library vision scoring;
- repeatable installation on every declared host;
- raw benchmark evidence behind every efficiency or quality claim;
- prominent gratitude and attribution to Brad Bonanno and
  `bradautomates/claude-video`.

## 2. Non-negotiable constraints

1. Preserve all four install surfaces:
   - Claude Code marketplace/plugin;
   - `npx skills add abe238/claude-video-plus -g`;
   - release-built `watch.skill` for claude.ai;
   - manual clone plus `skills/watch` symlink.
2. Keep `skills/watch/` self-contained and host-independent.
3. Keep successful repeat setup preflight silent and under 100 ms.
4. Add no mandatory model, embedding runtime, API key, vector database, or service.
5. Keep an executable Control Adapter until v1.0 passes its final gate.
6. Never send audio, cookies, keys, or private media to an Adapter the user did not
   enable or the documented `auto` policy did not select.
7. Automatic HTTP probing is loopback-only. Remote transcription endpoints require
   explicit configuration, TLS by default, and clear disclosure.
8. No default-path speed or reliability compromise is allowed. Measured jitter is not
   product latency; define the jitter allowance from no-op repeats before testing.
9. Optional Adapters may have measured tradeoffs only after an informed opt-in.
10. Do not publish a general superiority or novelty claim from the existing one-video
    benchmark.

## 4. Architecture

Use the repository's domain vocabulary from `CONTEXT.md`. Architecture suggestions use
Module, Interface, Implementation, Depth, Seam, Adapter, Leverage, and Locality.

### 4.1 Thin entry Module

`watch.py` parses user intent, selects Control or Candidate behavior, coordinates deep
Modules, and renders the final result. It must not become a switchboard containing the
Implementation of every source, transcription backend, retry, cache, or scorer.

The pipeline passes normalized results:

```text
Video source + Question + Evidence budget
  -> Acquisition result
  -> Transcript result
  -> Frame selection result
  -> Evidence manifest
  -> reader-ready evidence
```

Every result records non-secret provenance, attempted Adapters, selected Adapter,
warnings, failure class, and fallback reason.

### 4.2 Acquisition Module

One deep Interface owns local paths, yt-dlp, caption/metadata acquisition, and classified
recovery. Its result contains media path, subtitle candidates, metadata, source identity,
attempt records, selected strategy, and warnings.

Ordered Implementation:

1. Resolve a local Video source without network work.
2. Run current default yt-dlp behavior.
3. Classify YouTube/SABR/403/429/login/region/format failures.
4. Retry only eligible failures with tested player clients, one at a time.
5. Retry the normal quality selector with itag 18 as the **last** format fallback.
6. Apply configured browser cookies when explicitly enabled.
7. Preserve a clear unsupported/private/region-locked result when recovery is unsafe.

Do not copy Taelo's exact format string: placing `18/` first prefers 360p whenever it is
available. Port the resilience idea onto current upstream with tests and without losing
later security fixes.

Optional source Adapters such as Fathom may join this Seam only with a concrete use case,
tests, and no mandatory account dependency.

### 4.3 Transcription Module

One deep Interface normalizes every transcript source into timestamped segments with:

```text
start | end | text | language | Adapter | model | confidence? | warnings
```

Default `auto` order:

1. native captions;
2. same-basename local VTT/SRT sidecar;
3. configured loopback OpenAI-compatible endpoint;
4. YAP on supported macOS systems;
5. Groq Whisper;
6. OpenAI Whisper;
7. explicit no-transcript fail-open result.

Required optional configuration:

```text
WATCH_STT_ORDER=local-http,yap,groq,openai
WATCH_STT_URL=http://127.0.0.1:8082
WATCH_STT_MODEL=Systran/faster-whisper-medium
WATCH_LANGUAGE=auto
```

The CLI exposes explicit Adapter selection as well as `auto`. YAP and local servers are
detected, never silently installed. A successful earlier Adapter prevents sending audio
to later Adapters. Backend, model, timestamps, and fallback reasons enter the Evidence
manifest; keys and request headers never do.

### 4.4 Evidence state Module

One deep Interface owns resumable stage receipts, safe transcript/Scout-index reuse, and
portable Evidence bundles. It must not become an unbounded default media cache.

Minimum correctness and privacy requirements:

- atomic temporary write plus rename;
- schema and checksum validation;
- corrupt entries become cache misses;
- owner-only permissions;
- bounded size/TTL and explicit purge;
- duplicate-safe concurrent writes;
- keys include source identity, range, language, Adapter/model, policy, and schema;
- no source media or selected private frames persist by default;
- uninstall and purge distinguish configuration, evidence state, and optional runtimes.

Portable Evidence bundles use relative paths, checksums, tool/schema versions, Evidence
budget accounting, completeness state, and non-secret provenance. Export, verify, and
replay must be deterministic. Bundles never silently include source media, cookies,
keys, or unrelated local files.

### 4.5 Semantic relevance Seam

Deepen the Evidence compiler with a `Relevance scorer` Seam rather than building a
separate vector-database product. Keep every score and its provenance visible in the
Evidence manifest.

Adapters:

1. lexical relevance plus exact entity/number/timestamp guards (default);
2. optional local text-embedding relevance;
3. optional explicitly configured remote text-embedding relevance;
4. optional multimodal frame/OCR-to-Question relevance;
5. hybrid rank fusion combining independent lexical, semantic, coverage, novelty, and
   temporal-obligation signals.

The default hybrid policy is coarse-to-fine:

1. build dependency-free Evidence spans and coverage anchors;
2. run lexical/entity/numeric retrieval first;
3. invoke semantic scoring only when lexical evidence is weak, ambiguous, paraphrased,
   cross-lingual, or missing a required Question facet;
4. rerank or expand a bounded candidate set rather than embedding every frame eagerly;
5. retain exact-match guards and required before/after evidence regardless of semantic
   rank;
6. use the Control Adapter when semantic scoring is unavailable or insufficient.

Semantic indexes are an optional Scout-index representation, not a mandatory global
vector store. Cache keys include model/version, source fingerprint, text normalization,
language, and schema. Remote embedding calls require explicit disclosure and count all
tokens, latency, dollars, and transmitted text/images in total-system measurements.

Do not collapse all signals into an uninspectable confidence number. Use separately
reported scores and a versioned fusion policy, with reciprocal-rank fusion or another
calibration-resistant method as the initial Candidate.

### 4.6 Vision scoring Module

Keep vision scoring inside the existing FFmpeg and standard-library implementation
until measurements justify a genuinely different Adapter. Evaluate multi-scale local
change, temporal diversity, coverage, and FFmpeg-native scene/freeze signals on identical
timestamps and Evidence budgets. Do not add OpenCV or PySceneDetect; ADR-0001 records the
negative ablation and product decision.

### 4.7 Benchmark Module

The Benchmark Module runs the frozen Control and Candidate through the same cases,
reader-model epoch, Evidence budgets, prompts, retry policy, and grading protocol. It
produces raw JSONL results and a human-readable report. Development and confirmatory
data never share a source family.

## 5. Execution milestones

Each milestone lands in coherent, tested commits. Per-commit deterministic tests run
throughout; statistical evaluation occurs only at named development and confirmatory
gates.

### Milestone 0 — release integrity and canonical documentation

Deliverables:

- make this plan canonical after owner approval and mark older plans historical;
- correct the nonexistent single-test command in `AGENTS.md`;
- add push/PR CI for the supported Python/OS matrix;
- add isolated Agent Skills install, plugin-layout, symlink-layout, and bundle tests;
- fix subtree archive exclusions so `.skillignore` and `build-skill.sh` do not enter the
  runtime bundle unless intentionally retained;
- qualify repository/plugin claims as results from an initial one-video benchmark;
- state plainly that the multi-video battery is in progress;
- replace “byte-for-byte original behavior” with a tested control-equivalence claim;
- tag a maintenance release and publish a real `watch.skill` artifact;
- retain the dedicated Gratitude and Attribution section and upstream history.

Exit gate:

- all tests and installation smoke checks pass from a fresh clone;
- the release artifact contains exactly one `SKILL.md` and only intended runtime files;
- every public numeric claim links to raw data and states its population;
- no feature behavior changes in this milestone.

### Milestone 1 — acquisition resilience and caption coverage

Deliverables:

- normalized Acquisition result and failure taxonomy;
- current default yt-dlp path preserved as the first attempt;
- SABR/403 retry classification and bounded retry ladder;
- safe `WATCH_COOKIES_BROWSER` support with browser/profile validation;
- itag 18 as final fallback, never first choice;
- configurable subtitle language and ordered selection;
- JSON3/anti-429 caption fallback where supported;
- optional Fathom source Adapter behind an explicit capability flag;
- argv, error-classification, redaction, fallback-order, and failure-injection tests.

Exit gate:

- Control-equivalent behavior on successful ordinary downloads;
- zero cookie/key disclosure in logs, manifests, or exceptions;
- higher completion on the frozen acquisition-failure corpus;
- no median or p95 slowdown on ordinary successful acquisitions beyond measured no-op
  jitter;
- every fallback is visible and reproducible.

### Milestone 2 — pluggable local-first transcription

Deliverables:

- normalized Transcription result and Adapter probe/execute contract;
- sidecar VTT/SRT discovery before ASR;
- loopback OpenAI-compatible Adapter, verified against the local `:8082` server shape;
- YAP Adapter using VTT or timestamped JSON output;
- Groq/OpenAI moved behind the same Seam;
- explicit Adapter selection plus configurable `auto` order;
- setup status reports available local/cloud choices without nagging or installation;
- language selection, timestamp normalization, partial-failure, and fallback tests;
- JSON output for machine-readable pipeline diagnostics.

Exit gate:

- each Adapter passes accuracy, timestamp, latency, memory, privacy, and failure tests;
- local HTTP auto-probe rejects non-loopback targets;
- YAP absence and local-server failure are silent during successful fallback;
- captions/sidecars prevent unnecessary local or cloud ASR;
- no new mandatory dependency or key.

### Milestone 3 — range-aware audio, safe resume, and reuse

Deliverables:

- extract only requested audio ranges before any ASR upload or local inference;
- restore absolute source timestamps after range transcription;
- silence-aware chunk boundaries where they improve accuracy or retry cost;
- bounded retries and completed-chunk receipts;
- safe transcript/Scout-state reuse under the Evidence state Module;
- `--fresh`, status, verify, and purge behavior;
- cold, same-task follow-up, and new-task warm measurements.

Exit gate:

- focused requests process no audio outside the requested interval plus a documented
  codec boundary allowance;
- timestamp accuracy remains within the preregistered bound;
- retries never reprocess successful chunks unnecessarily;
- corruption, stale schema, interruption, concurrency, and permission tests pass;
- default private-media retention does not increase.

### Milestone 4 — portable evidence and deeper compilation

Deliverables:

- versioned portable Evidence manifest and bundle schemas;
- export, verify, and replay operations;
- Question and Evidence budget captured without shell-quoting ambiguity;
- explicit Control and Candidate selection with recorded fallback reasons;
- progressive “retrieve, verify sufficiency, expand only affected Evidence spans” flow;
- frame/table mining and conflicting-claim reconciliation preserved;
- replay tests prove that the same bundle produces the same reader-ready evidence;
- repeated questions can reuse a verified Scout index without reopening all media.

Exit gate:

- Control fallback remains executable from the installed skill;
- bundle verification detects alteration, missing files, schema mismatch, and unsafe
  absolute paths;
- no secret or unrequested media is exported;
- targeted and Coverage question behavior meet their separate development gates;
- total system input is reported so work cannot be hidden in selector calls.

### Milestone 5 — hybrid semantic evidence retrieval

Deliverables:

- frozen paraphrase, concept, cross-lingual, visual-text, exact-number, and exact-entity
  fixtures that lexical-only retrieval can and cannot solve;
- dependency-free lexical scorer retained as the default Adapter and exactness guard;
- versioned relevance-score records in every Evidence manifest;
- optional local text-embedding Adapter in an isolated runtime or configured local
  endpoint;
- optional remote embedding Adapter behind explicit configuration and privacy notice;
- bounded semantic reranking after lexical/coverage candidate generation;
- uncertainty/facet trigger deciding when semantic work is worth its cost;
- hybrid rank fusion with independent lexical, semantic, coverage, novelty, and
  temporal-obligation ablations;
- optional OCR/frame-text semantic Adapter evaluated separately from text-only
  embeddings;
- safe Scout-index invalidation and reuse under the Evidence state Module.

Semantic promotion gate:

- higher Evidence-span recall on paraphrase/concept/cross-lingual cases;
- zero regression on exact name, number, timestamp, negation, and before/after fixtures;
- non-inferior answer quality overall and by supported Question class;
- total-system tokens, embedding work, latency, memory, network, and dollars included;
- semantic calls occur only when the versioned trigger predicts measurable Leverage;
- no private transcript or frame is sent remotely without explicit configuration;
- a local/remote Adapter failure falls back deterministically to lexical-plus-coverage;
- a model or fusion term that does not improve the Pareto frontier is removed.

Exit gate:

- publish lexical-only, semantic-only, and hybrid ablations on identical cases;
- document which Question classes benefit and route semantic work only to those classes;
- keep lexical-only operation fully supported with no model download or API key.

### Milestone 6 — dependency-free vision selection

Deliverables:

- stronger dependency-free multi-scale and temporal-diversity scoring;
- identical candidate timestamps and Evidence budgets for every candidate policy;
- separate ablations for current 16×16 scoring and FFmpeg/standard-library candidates;
- local-change, UI, slides, gradual-transition, motion, before/after, and static-video
  fixtures;
- documented CPU, memory, latency, false-drop, event-recall, duplicate, and token data;
- deterministic fallback to the current scorer.

Vision-policy promotion gate:

- zero known false drops on deterministic required-change fixtures;
- non-inferior Evidence-span recall and answer quality;
- a statistically credible improvement in at least one primary outcome: key-event
  recall, duplicate-reader-image reduction, or reader tokens;
- no default-path speed, reliability, or installation regression;
- the gain remains after counting all decode, scoring, and selection work;
- if the gain is class-specific, route only that Question/video class to the candidate;
- the current scorer remains available until a candidate beats the total-system Pareto
  frontier on the declared platforms.

Exit gate:

- publish the complete ablation, including negative results;
- delete any scorer or signal that does not improve the Pareto frontier;
- do not advertise “enhanced recognition” until this gate passes.

### Milestone 7 — v1.0 confirmation and distribution

Deliverables:

- complete multi-video development battery across independent source families,
  duration buckets, languages, caption states, and Question classes;
- validate the semantic grader against dual human judgments before relying on it;
- use the pilot to determine powered sample sizes and freeze the analysis plan;
- freeze one Candidate commit/configuration before opening confirmatory data;
- run exactly one untouched confirmatory Tier 2 evaluation;
- run external public benchmark subsets only for a published-baseline claim;
- independent fresh-clone reproduction;
- complete install/update/uninstall matrix;
- v1.0 changelog, limitations, attribution, raw data, tag, and release artifact.

Final release gate:

- answer quality clears preregistered non-inferiority overall and by supported class;
- Evidence-span recall and required before/after recall clear their bounds;
- completion rate is at least the Control rate in every supported environment;
- targeted questions reduce median reader input by at least 50%;
- Coverage questions reduce median reader input by at least 25% without lost coverage;
- all-model input and total system work are reported alongside reader-only savings;
- default-path median and p95 latency are no slower than Control beyond measured no-op
  jitter in each duration bucket;
- successful repeat preflight remains under 100 ms;
- install artifacts remain under 250 KB compressed with no bundled wheel/model/binary;
- all supported install, update, invocation, rollback, purge, and uninstall tests pass;
- limitations and every underpowered or regressed class are disclosed;
- no public claim exceeds what the confirmatory population supports.

## 6. Measurement protocol

### 6.1 Corpora

- Tier 1: deterministic generated/curated fixtures, run on every commit.
- Tier 2 development slice: reusable during milestones, partitioned by highest
  dependence unit (video, derivatives, series/channel/source family).
- Tier 2 confirmatory reserve: unseen source families; sample size drawn after the pilot
  from a preregistered frame without inspecting outcomes.
- Tier 3: optional external-validity evaluation after Candidate freeze; never used to
  tune a release evaluated on the same items.

If a failed confirmation influences implementation, retire that confirmatory cohort and
draw a new sealed cohort before making a new claim.

### 6.2 Required Question classes

- transcript-only;
- visual-only;
- cross-modal;
- exact timestamp;
- temporal order and before/after;
- targeted retrieval;
- coverage summary;
- visible text/table/UI;
- motion and gradual transition;
- multilingual and caption-less;
- unanswerable/insufficient evidence;
- hype/padding exclusion.

### 6.3 Primary measurements

- answer accuracy and citation support;
- Evidence-span recall and temporal IoU;
- required before/after recall and false-drop rate;
- completion and categorized failure rate;
- reader text/image/total input;
- all-model input and total system cost;
- lexical, semantic, and hybrid retrieval recall by Question class;
- embedding calls, embedded units, cache hit rate, model/version, and rerank latency;
- acquisition bytes and attempts;
- ASR WER, timestamp error, and real-time factor;
- median and p95 latency by cold/warm state and duration bucket;
- peak memory, CPU, disk, network, and optional initialization cost;
- install/update/uninstall success and artifact size.

The cheapest applicable Control mode is a preregistered deterministic mapping using only
the Question, explicit user hints, and source metadata—never gold Evidence spans.

## 7. Rejected dependency: OpenCV

OpenCV is not part of v1.0. On the real 38-minute development video, the composite
OpenCV prototype recovered 5 of 12 previously curated screen moments versus 6 of 12 for
the current 16×16 scorer, produced one additional near-duplicate pair, used the same
estimated reader-image tokens, and took about 304 ms to score versus about 64 ms. A
synthetic six-event fixture also showed no recall gain at the shared ten-frame budget.

Those development results are not a general scientific conclusion, and the 12 moments
are a proxy rather than an independent gold set. They are sufficient for the product
decision because OpenCV showed no total-system benefit while adding a large wheel,
startup memory, ABI/platform maintenance, and a new failure surface. Remove the Adapter,
installer choice, environment variable, and PySceneDetect policy from v1.0. Reopen only
with materially new evidence and explicit owner approval. The complete result and its
limitations are in `docs/benchmarks/2026-07-11-opencv-ablation/`.

## 10. Approval and execution contract

Approval of this plan authorizes implementation in milestone order, with direct commits
to `main` at coherent tested checkpoints as previously requested. It does not authorize:

- publishing the private repository;
- making unsupported superiority or novelty claims;
- installing optional dependencies globally on the user's machine;
- sending private media to new remote providers;
- skipping a failed gate to preserve schedule.

After approval, execution begins with Milestone 0. Each milestone handoff reports:

- commits pushed;
- tests and environments run;
- raw benchmark or failure-injection artifacts;
- deviations from this plan;
- remaining risks and the next gate.
